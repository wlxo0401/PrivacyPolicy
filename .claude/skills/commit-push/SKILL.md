# Commit & Push Skill

변경사항을 분석하고, 한국어 커밋 메시지를 작성한 후, 커밋하고 푸시합니다.

## Procedure

### Step 1: 변경사항 확인

다음 명령어를 병렬로 실행하여 현재 상태를 파악합니다:

- `git status` — 변경/추가/삭제된 파일 목록 확인
- `git diff` — unstaged 변경 내용 확인
- `git diff --cached` — staged 변경 내용 확인

변경사항이 없으면 "커밋할 변경사항이 없습니다." 를 출력하고 종료합니다.

### Step 2: 커밋 분리 & 메시지 초안 & 확인

변경사항을 분석하여 **성격이 다른 변경은 별도 커밋으로 분리**합니다.

#### 커밋 분리 규칙

- **하나의 커밋 = 하나의 목적**. 다른 기능, 다른 버그, 다른 목적의 변경은 반드시 별도 커밋으로 나눕니다.
- 제목에 "—"나 ","로 여러 내용을 나열하지 않습니다.
- 분리 기준: 기능 추가 vs 버그 수정 vs 리팩토링 vs 스타일 변경 등 목적이 다르면 분리합니다.
- 같은 목적의 관련 변경은 하나의 커밋으로 묶어도 됩니다.

#### 커밋 메시지 형식 규칙

**제목 (반드시 `- `로 시작):**
```
- {type}: {한국어 설명}
```

> ⚠️ **주의**: 제목은 반드시 `- ` (대시+공백)으로 시작해야 합니다. 절대 생략하지 마세요.
>
> ❌ Bad: `feat: 기능 추가`
> ✅ Good: `- feat: 기능 추가`

**body (description):**
- 변경 내용을 `1. 2. 3.` 번호 매김으로 구체적으로 정리합니다.
- AI가 작업한 내용을 사용자가 한눈에 파악할 수 있도록 기술합니다.

**type 종류:**

| Type | 의미 |
|------|------|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `chore` | 빌드, 설정, 의존성 등 기능 외 변경 |
| `refactor` | 리팩토링 (기능 변경 없이 코드 개선) |
| `style` | UI/코드 스타일 변경 |
| `docs` | 문서 변경 |
| `perf` | 성능 개선 |
| `test` | 테스트 추가/수정 |

**예시 (단일 커밋):**
```
- fix: 종베 스캔 실패 종목 필터링 버그 수정

1. scanResults에서 시세 조회 실패한 종목을 제외하는 guard 추가
2. nil price 종목이 스코어링에 포함되던 문제 해결
```

**예시 (여러 커밋으로 분리하는 경우):**

커밋 1:
```
- fix: 종베 스캔 실패 종목 필터링 버그 수정

1. scanResults에서 시세 조회 실패한 종목을 제외하는 guard 추가
2. nil price 종목이 스코어링에 포함되던 문제 해결
```

커밋 2:
```
- style: 종베 스캔 결과 색상을 한국 관례에 맞게 적용

1. 상승 종목 색상을 빨강(#EF4444)으로 변경
2. 하락 종목 색상을 파랑(#3B82F6)으로 변경
```

`AskUserQuestion` 으로 사용자에게 메시지를 보여주고 확인을 요청합니다:

- **커밋이 1개인 경우**: "다음 커밋 메시지로 커밋하시겠습니까?\n\n`{메시지 초안}`\n\n수정이 필요하면 원하시는 메시지를 입력해주세요."
- **커밋이 여러 개인 경우**: "다음과 같이 커밋을 분리하여 진행하시겠습니까?\n\n**커밋 1:**\n`{메시지}`\n- 파일: {파일 목록}\n\n**커밋 2:**\n`{메시지}`\n- 파일: {파일 목록}\n\n수정이 필요하면 원하시는 내용을 입력해주세요."
- 사용자가 승인하면 해당 메시지 사용
- 사용자가 수정 메시지를 제공하면 수정된 메시지 사용

### Step 4: 순차 커밋 & 푸시

분리된 커밋별로 관련 파일만 staging → commit을 반복하고, 모든 커밋 완료 후 한 번에 push합니다.

**각 커밋마다:**
1. **파일 스테이징**: 해당 커밋에 속하는 파일만 개별적으로 `git add` 합니다.
   - `.env`, `credentials`, 시크릿 파일 등 민감한 파일은 **절대 커밋하지 않고** 경고를 출력합니다.
2. **메시지 검증**: 커밋 메시지 제목이 `- `(대시+공백)으로 시작하는지 반드시 확인합니다. 누락된 경우 추가합니다.
3. **커밋 실행**: `git commit` 으로 커밋합니다.
   - 제목과 body를 포함한 커밋 메시지를 작성합니다.
   - 커밋 메시지 끝에 `Co-Authored-By: {현재 사용 중인 Claude 모델명} <noreply@anthropic.com>` 헤더를 추가합니다. (예: `Claude Opus 4.6 (1M context)`, `Claude Sonnet 4.6`, `Claude Haiku 4.5` 등 실제 모델에 맞게 작성)
   - HEREDOC 형식으로 메시지를 전달합니다.

**모든 커밋 완료 후:**
4. **리모트 동기화**: `git pull --rebase` 를 실행하여 리모트의 최신 변경사항을 먼저 반영합니다.
   - rebase 충돌이 발생하면 `git rebase --abort` 로 중단하고, 충돌 내용을 사용자에게 알려준 뒤 종료합니다.
5. **푸시 실행**:
   - 현재 브랜치의 remote tracking 상태를 확인합니다.
   - tracking이 설정되어 있지 않으면 `git push -u origin {branch}` 로 푸시합니다.
   - tracking이 설정되어 있으면 `git push` 로 푸시합니다.
6. **결과 출력**: 모든 커밋 해시와 푸시 결과를 출력합니다.

## Notes

- 모든 출력은 한국어로 작성합니다.
- `.env`, `credentials.json`, `.secret` 등 민감 파일은 커밋하지 않고 경고합니다.
- `git add -A` 나 `git add .` 대신 개별 파일을 명시적으로 추가합니다.
- force push는 절대 사용하지 않습니다.
- 커밋 메시지는 HEREDOC 형식으로 전달하여 포맷을 보존합니다.
